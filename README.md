# OCR Restaurant Receipt Processor

A serverless automated pipeline designed to bridge physical restaurant receipts with cloud-based accounting systems. The system ingests receipt images via a Telegram bot, applies spatial Optical Character Recognition (OCR), resolves conversational edge cases, and logs the parsed financial data into Google Sheets in real-time.

## Architecture Overview

The system is built on a serverless architecture optimized for high-concurrency, fast deployment, and minimal operational overhead:

- **Frontend Interface:** Telegram Bot API operating via Webhooks.
- **Backend Compute:** Modal (Serverless containerized Python execution).
- **Computer Vision / OCR:** PyTesseract (Tesseract OCR) combined with Pillow for dynamic image preprocessing filters (grayscale, binarization, sharpening).
- **State Management:** Modal Dict functioning as a persistent, low-latency key-value store for asynchronous conversational states.
- **Database / Logging:** Google Sheets API via gspread.

## Key Features

- **Spatial Document Understanding:** Replaces traditional global regular expressions with coordinate-based bounding box parsing. The system dynamically reads the horizontal alignment to distinguish the main receipt body from secondary merchant vouchers, effectively eliminating data contamination.
- **Resilient Data Extraction:** Integrates Longest Common Subsequence (LCS) algorithms and Levenshtein edit distance to validate waitstaff names against a strict organizational catalog. Incorporates a strict confidence threshold to block false positives generated by OCR noise.
- **Conversational State Machine:** Identifies ambiguous financial data (e.g., missing tip payment methods) and interactively prompts the user for clarification. The context of the pending operation survives container spin-downs, reliably matching the user's asynchronous reply with the correct in-flight receipt.
- **Idempotency Controls:** Neutralizes duplicate payloads and timeout-driven network retries from the Telegram API to guarantee single-source-of-truth accounting entries.
- **Multi-Scenario Routing:** Evaluates complex split-payment inputs to autonomously distribute values into the correct database tables (e.g., allocating a credit card base payment to the main ledger while shifting a cash tip into a separate parallel ledger).

## Development Setup

1. **Environment Configuration**
   Create a Python virtual environment and install the required dependencies:
   ```bash
   python -m venv .venv
   source .venv/bin/activate
   pip install fastapi gspread google-api-python-client google-auth pytesseract Pillow requests modal
   ```

2. **Secrets Management**
   The application secures environment variables using Modal Secrets. You must provision your Telegram Bot Token and your Google Service Account payload:
   ```bash
   modal secret create castillo-bot-secrets \
     TELEGRAM_BOT_TOKEN="your_token" \
     GOOGLE_SERVICE_ACCOUNT_JSON='{...}'
   ```

3. **Running Locally**
   Start the development instance with hot-reloading enabled:
   ```bash
   modal serve src/main.py
   ```

## Deployment

Deploy the system to production on Modal's edge infrastructure with a single command:

```bash
modal deploy -m src.main
```
